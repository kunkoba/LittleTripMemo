◆◆アプリ名：LittleTripMemo（小さな旅の思い出）

■技術スタック 
・Framework：ASP.NET Core MVC (.NET 8/9) 
・DB：PostgreSQL (EF Core) 
・JWT
・CSS：Tailwind CSS (CSSファイルは極力触らない方針) 
・Map：Leaflet

■DB/モデル設計（Spotsテーブル） 
・Id (UUID)：オフライン保存を考慮 
・UserId (string)：ユーザー識別 
・Latitude/Longitude (double)：座標 
・TargetAt (DateTime)：メモ発生時または予定日時 
・IsPlan (bool)：メモ(false) / 計画(true) の判別用 
・IsPublic (bool)：公開設定

●●サーバ側の基本構造（覚えといて）
・コントローラが画面単位で、画面から要望を受け付けするところ
・サービスはユースケース単位で、業務ロジックはここに詰めるがそれ以外はもたない
　トランザクション管理、リクエスト、レスポンスの型はここで持つ
・モデルはスネークケースでテーブルと同型で用意済み
　サービスでリクエスト・レスポンスのマッピング
・リポジトリはテーブル単位で、データを直接さわるところ
・リポジトリの基底クラスで、SQLの操作、エラーハンドリング、ログ出力
・主要テーブルはDapper、アカウントはEF（規定の機能）

/**********************************************************************
 * LittleTripMemo - フロントエンド設計整理メモ
 * 対象：Map / Dialog / UI 制御
 * フェーズ：問題洗い出し・方針確定（実装前）
 * 目的：中断・再開を容易にするための記録
 **********************************************************************/

/*
============================================================
■ 背景・目的
============================================================

- Map 関連モジュールが肥大化（約2000行）
- Dialog 生成・UI ロジック・業務処理が混在
- 責務分離と疎結合化が急務

本メモは、設計方針の合意内容を固定化するためのもの。
実装はまだ行わない。
*/


/*
============================================================
■ 現状の問題点
============================================================

1. Map / Dialog / UI ロジックが同一モジュールに混在
   - 追跡困難
   - 影響範囲が広い

2. ダイアログごとに個別実装が乱立
   - HTML生成
   - Bootstrap制御
   - イベント設定
   - Promise管理
   が各関数に分散

3. 呼び出し方式が混在
   - 直接関数呼び出し
   - Promise
   - グローバル依存
*/


/*
============================================================
■ 決定した基本方針
============================================================

【方針① Promise と Event の使い分け】

▼ Promise を使うケース
- ユーザーの「返事」が必須
- 結果がないと次に進めない

例：
- OK / Cancel ダイアログ
- 入力フォーム
- 選択ダイアログ

→ 「返事がないと処理不能」


▼ CustomEvent を使うケース
- 画面を開きたいだけ
- 無視されても業務的に破綻しない

例：
- 詳細パネルを開く
- 編集画面を表示
- Map → UI への通知

→ 「要求を投げるだけ」
*/


/*
============================================================
■ 設計思想（重要）
============================================================

- 疎結合を最優先
- 呼び出し元は「誰が処理するか」を知らない
- 受け手は「誰が発火したか」を気にしない

Event は「UIインターフェース」として扱う
*/


/*
============================================================
■ 構造方針（合意済み）
============================================================

【DialogService を導入する】

責務：
- ダイアログHTML生成
- Bootstrap modal 管理
- 多重モーダル制御
- Promise / Event の窓口

呼び出し側（Map / UI）は：
- ダイアログ構造を知らない
- Event 発火 or Promise 呼び出しのみ
*/


/*
============================================================
■ 構造イメージ
============================================================

Map / UI
 ├─ CustomEvent（表示要求）
 └─ Promise（結果が必要な場合）

DialogService
 ├─ Event Listener
 ├─ Dialog生成
 └─ 結果返却（Promise）
*/


/*
============================================================
■ UI設計に関する合意
============================================================

- Map は全画面
- マーカークリック → 最小ポップアップ
- 詳細操作は「詳細パネル」に集約
- ポップアップはトリガー専用

目的：
- UI複雑化の防止
- Mapの責務を軽く保つ
*/

============================================================
■ 現フェーズの位置づけ
============================================================

- 実装しない
- 問題洗い出し
- 方針確定
- 文書化

→ いつでも中断・再開可能な状態を作る
*/
