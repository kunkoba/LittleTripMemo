# アプリ開発支援 指示書：LittleTripMemo

## 1. プロジェクト概要
- **アプリ名:** LittleTripMemo
- **目的:** 地図とメモを統合管理するアプリ。地点情報に紐づいたメモを管理・表示する。
- **現在の状況:** 
  - サーバーサイド処理（ASP/Postgres）はある程度完成。
  - 現在、ローカル（フロントエンド）側を集中開発中。
  - サーバーとのAPI連携部分はこれから実装予定。

## 2. 技術スタック（移植先 / 開発中）
- **言語:** ASP.NET (Core想定), JavaScript (Vanilla JS), HTML5
- **主要ライブラリ:** 
  - **地図:** Leaflet.js（ファイル構成より推測）
  - **CSS:** Tailwind CSS, 自作CSS（app-style, component-style）
  - **DB:** PostgreSQL (Server-side), LocalStorage/IndexedDB (Client-side想定)
- **インフラ:** Docker

## 3. ディレクトリ構造（wwwroot配下）
※AIはこの構造を前提にファイル間の参照（import/パス指定）を提案すること。
- `/` : index.html (メイン)
- `/component/` : UI用HTML部品（fatal-error, template）
- `/css/` : スタイルシート（app, component, leaflet）
- `/img/` : 画像リソース（face, weather）
- `/js/` : メインロジック
  - `app-manager.js` (全体管理), `common.js` (共通関数), `local-db-controller.js` (DB操作) 等
- `/js/component/` : UI部品専用のコントローラー
  - `map-controller.js`, `marker-controller.js`, `ui-manager.js` 等

## 4. コーディング規約（最優先）
- **命名規則:** 
  - Coreロジックのメソッド: `camelCase`
  - Controllerクラス/オブジェクト: `PascalCase`
  - DBフィールド名: `snake_case` (Postgres準拠)
  - ※既存コードにばらつきがある場合は、解析した上で最適な着地点を提案すること。
- **記述スタイル:** 
  - **空行禁止:** コード内に空行を入れない。視覚的な区切りが欲しい場合は `//` だけの行を入れる。
  - **複数行コメント禁止:** `/* ... */` は使用せず、すべて `//` で記述する。
  - **簡潔かつ明解:** 短いコードを好むが、初心者にも理解できる範囲のモダンさ（ES6+）に留める。
- **エラー処理:** 
  - 開発段階では複雑なtry-catchは不要。ロジックの完成を優先し、後にまとめて組み込む。

## 5. コミュニケーション・支援方針
- **回答形式:** 
  - **結論から述べる（重要な結論は太字にする）。**
  - 説明は簡潔に。詳細な解説が必要な場合はこちらから要求する。
- **アドバイス:** 
  - イエスマンにならず、設計上の懸念、バグの予兆、より良い代替案があれば積極的に「警告・提案」すること。
  - 実装者のレベル（現在向上中）に合わせ、「なぜそうするのか」という教育的視点を適宜含める。
- **コード維持:** 
  - 既存のコメントは最大限尊重し、勝手に削除しない。
  - セクション分けのコメントを適切に付与し、見通しを良くする。

## 6. AIへの具体的な依頼事項
1. 既存コードの解析とレベル感の把握。
2. 不整合（変数名の矛盾や依存関係のミス）の指摘。
3. 未実装部分（サーバー連携やローカルDB操作）の段階的な実装サポート。
4. UI/UX向上のためのTailwind CSS活用アドバイス。
